# お知らせ文章作成スキル

BUZZ BASEの「運営からのお知らせ」（`notice-from-management`）に投稿する文章とタイトルを生成するスキル。

PR・機能内容を受け取り、ユーザー視点に翻訳した日本語のお知らせを作成し、Web版・Mobile版の両方で正しくレンダリングされるマークダウン形式で出力する。

## 入力

以下のいずれかを受け取る:

- GitHub PRのURL（例: `https://github.com/ippei-shimizu/buzzbase_mobile/pull/72`）
- リポジトリ名 + PR番号（例: `buzzbase_mobile #72`）
- 機能の説明（PRが無い場合）

複数PRをまたぐ機能の場合は、すべてのPR情報を統合する。

## ワークフロー

### 1. PR情報の取得

引数からPR URLまたは番号を抽出し、`gh pr view` でPRの本文・差分・コミットを取得する。

```bash
gh pr view <PR番号> --repo <owner/repo>
```

リポジトリは以下から判定する:

| URL/手がかり | リポジトリ |
| ---- | ---- |
| `buzzbase_mobile` | `ippei-shimizu/buzzbase_mobile` |
| `buzzbase_front` | `ippei-shimizu/buzzbase_front` |
| `buzzbase_back` | `ippei-shimizu/buzzbase_back` |
| 指定なし | ユーザーに確認、または機能内容から推定 |

PRの「実装概要」「背景」「受入基準」を中心に読み、**ユーザーが体感する変化**を抽出する（技術的な実装詳細は捨てる）。

### 2. お知らせのタイプを判定

過去のお知らせから以下のパターンを使い分ける:

| タイプ | 用途 | タイトル例 |
| ---- | ---- | ---- |
| 新機能 | 機能追加 | `〇〇できるようになりました` / `〇〇機能を追加しました` |
| アプリリリース | 新プラットフォーム展開 | `BUZZ BASEのiOSアプリをリリースしました` |
| お願い | フィードバック募集等 | `アイデア・ご要望を募集します` |
| 不具合報告 | 障害・対処方法 | `〇〇による不具合` |

### 3. タイトルの生成

以下のルールでタイトルを作成する:

- **「何ができるようになったか」が一目でわかる形式**を優先（「〇〇機能を追加しました」「〇〇できるようになりました」）
- 過度な感嘆符・絵文字は使わない
- 40文字以内を目安に簡潔に
- 対象ユーザーが検索でヒットする語を含める

### 4. 本文の生成

過去お知らせのテンプレートに沿って、以下の構成で生成する:

```markdown
いつもBUZZ BASEをご利用いただきありがとうございます。

<冒頭1〜2文で、何ができるようになったかを要約>

**どんな機能？**

<機能の概要を2〜4文で説明>

<必要に応じて箇条書きで選択肢・項目を列挙>
- 項目1
- 項目2

**使い方**

<どの画面・どの導線から使えるかを明示し、操作手順を番号付きリストで記載>

1. 〇〇画面から「△△」をタップ
2. △△を入力／選択
3. 保存

<初期値・自動セット仕様がある場合は別小見出しを立てる>

**初期値について**

- 新規利用時の挙動
- 編集時の挙動
- ユーザーが「毎回操作する必要があるか」が分かるように書く

**主な変更点** または **できること**

- ユーザーが操作する画面・手順の変化
- 表示の変化
- 自動で行われる処理

<必要に応じて関連する追加要素（ツールチップ・補助画面・計算式説明など）を別小見出しで>

**〇〇が見られるようになりました** など

- 追加された画面や表示
- どこで確認できるか（画面名を列挙）

<必要に応じて画像>
[![Image from Gyazo](画像URL)](GyazoページURL)

**ユーザーへの影響**

<ユーザーの利用パターンごとに、具体的に何が変わるかを箇条書き>

- **〇〇している方**: どう変わるか、何を操作する必要があるか
- **△△している方**: 影響なし／自動で適用される、など
- **過去に記録したデータ**: どう扱われるか

**こんな方におすすめ**

- 対象ユーザー1
- 対象ユーザー2
- 対象ユーザー3

<既存データへの影響がある場合は「ユーザーへの影響」に統合する。重複は避ける>

ご意見・ご要望がありましたらお気軽にお問い合わせください。
- お問い合わせフォーム : [https://buzzbase.jp/contact](https://buzzbase.jp/contact)
- XのDM : [https://x.com/ippei_111](https://x.com/ippei_111)

**★ App Storeでのレビューにご協力ください**

BUZZ BASEを気に入っていただけましたら、App Storeでの評価・レビューにご協力いただけると大変励みになります。
今後とも、皆さまの声を元に新規機能・改善に努めてまいります。

[App Storeでレビューを書く](https://apps.apple.com/app/id6761011816?action=write-review)

今後ともBUZZ BASEをよろしくお願いいたします。
```

#### 文章作成ルール

- **ユーザー視点に翻訳する**: 「`appearance_type` カラムを追加」ではなく「代打・代走の試合を記録できる」と書く
- 一人称は使わず、「BUZZ BASE」を主語にする
- セクション見出しは `**太字**` で統一（`##` は使わない／装飾の階層を Web/Mobile 両方で安定させるため）
- 技術用語・カラム名・PR番号は出さない
- 関係ないセクションは省略してよい（例: 不具合報告に「こんな方におすすめ」は不要）
- 「★ App Storeでのレビューにご協力ください」と締めの挨拶は**必ず**含める

#### 「使い方」セクションのルール

- **必ず画面名・導線を明示する**（「マイページの〇〇ボタンから」「Step1の試合情報入力画面で」など、ユーザーが迷わず辿り着ける具体性）
- 手順は番号付きリスト、3〜5ステップ程度に収める
- 関連する画面が複数ある場合（ツールチップ・成績表示の追加先など）は別小見出しで画面名を列挙する
- PR本文の「実装詳細」「影響範囲」セクションから**ユーザーが触る画面のみ**を抽出する（裏側のコンポーネント名・サービス名は書かない）

#### 「ユーザーへの影響」セクションのルール

- **ユーザーの利用パターン別に分けて書く**（「〇〇している方」「△△している方」「過去にデータを記録した方」など）
- 各パターンで「何が変わるか」「操作が必要か」「具体例（数値・表示）」を1〜2文で
- 「変化なし」のパターンも明示する（不安を取り除くため）
- 過去データの扱いは必ず触れる（後方互換性への安心感を与える）

### 5. マークダウン記法の制約（重要）

**Mobile版の `markdownToHtml`（`mobile/utils/markdownToHtml.ts`）は裸URLを自動リンク化しない**。Web版（ReactMarkdown + remark-gfm）はGFMで自動リンク化されるが、Mobile版で表示されない事故を防ぐため**すべてのリンクを明示記法で書く**。

#### 必ず使う記法

| 用途 | 書き方 |
| ---- | ---- |
| リンク | `[表示テキスト](URL)` ← 裸URLは禁止 |
| 画像 | `![alt](画像URL)` |
| 画像リンク | `[![alt](画像URL)](リンク先URL)` |
| 太字 | `**text**` |
| 斜体 | `*text*` |
| 見出し | `# ` `## ` `### ` |
| 水平線 | `---` |
| リスト | `- item` |

#### 禁止事項

- 裸URLをそのまま貼る（`https://example.com` 単体）
- `<a href="...">` などの生HTML
- 表記法（`|---|---|`）— Mobile版は対応していない可能性があるため避ける
- ネストしたリスト（`  - subitem`）— Mobile版の自作パーサで崩れる可能性

### 6. 出力

生成したタイトルと本文を、コピペで管理画面にそのまま貼れる形でユーザーに提示する:

```
■ タイトル
〇〇できるようになりました

■ 本文
（マークダウン本文）
```

ファイルへの書き込みは行わない（管理画面で直接入力するため）。

## 過去お知らせの参照

過去の投稿は `back` の `management_notices` テーブルに保存されている。トーン・構成の参照が必要な場合は以下を確認する:

- `front/app/(app)/notice-from-management/notice-2024-03-12/_components/NoticeContent.tsx` — 旧形式の不具合報告例
- 管理画面（`/admin-management-console/notices`）の既存投稿

## 関連リンク（テンプレートで使う固定URL）

| 種別 | URL |
| ---- | ---- |
| お問い合わせフォーム | `https://buzzbase.jp/contact` |
| X DM | `https://x.com/ippei_111` |
| iOSアプリ | `https://apps.apple.com/jp/app/buzz-base/id6761011816` |
| App Storeレビュー | `https://apps.apple.com/app/id6761011816?action=write-review` |

App StoreのApple IDは `mobile/constants/appStore.ts` の `IOS_APPLE_ID`（`6761011816`）と一致させること。

## 注意事項

- 対話での確認は行わず、即座に生成して提示する
- 画像URL（Gyazo等）はユーザーから明示されない限り埋め込まない（プレースホルダも置かない）
- 推測で機能を盛らない（PRに書かれていない仕様は書かない）
- iOS版限定の機能の場合は本文中に明記する（Android版未対応の場合は「Android版については順次対応していきます」を添える）
