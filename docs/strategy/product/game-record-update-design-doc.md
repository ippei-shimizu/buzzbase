**作成日**: 2026-05-17
**ステータス**: ドラフト（壁打ち結果まとめ）
**関連ドキュメント**:

## 1. 概要

試合記録 / 打撃結果記録の体験と分析機能を大幅にアップデートする。打席ごとに詳細データを記録できるようにし、より深い自己分析を可能にする。一部の分析機能は Pro プラン限定とする。

### 主な変更点

- 試合記録の入力フローを3パターンに分岐（打撃のみ / 投手のみ / 両方）
- 球場マスタを新規追加（任意記録）
- 打撃結果記録を1打席ずつステップ式UIに刷新
- 打球方向をグラウンドイラストの絶対座標でタップ記録
- 任意の詳細データ（カウント、ランナー、打球の質、タイミング、球種、メモ）を追加
- 分析機能を拡張（打球分布の詳細化、状況別打率、打球の質グラフ等）
- 投手結果は変更なし

### 開発方針

- **mobile 先行**、Web 後追い
- 既存データには影響を与えない（既存マスタID・カラム保持）
- 既存集計テーブル `batting_average` は after_commit で自動再計算
- **既存ユーザーの成績・集計データがずれない / 消えないことを、各実装フェーズで必ず検証しながら進める**（詳細は §9.4 参照）

## 2. スコープ

### 含む

- 試合記録の入力フロー変更（3パターン分岐）
- 球場マスタの新規追加
- 打席ごとの詳細データ記録（打球方向の絶対位置・カウント・ランナー・打球の質・タイミング・球種・メモ・イニング）
- 新規マスタテーブル（球場・球種・打球の質・タイミング・打球の深さ）
- 分析機能の拡張（A〜F系の各種分析）
- Pro / 無料の機能切り分け
- mobile アプリへの実装

### 含まない（本Doc範囲外）

- Web版（front/）への移植は別フェーズ
- 投手結果記録の変更
- Pro プランの課金フロー（既存 PRD に委ねる）
- 球場マスタの admin 管理画面

## 3. データモデル

### 3.1 既存テーブルの拡張

#### `plate_appearances` テーブル

既存テーブルにカラムを追加（新テーブルは作らない）。既存データは新規カラム NULL のまま残す。

```
# 既存カラム（変更なし）
id
game_result_id (FK)
user_id (FK)
batter_box_number       # 第何打席か
batting_result          # 表示用テキスト（"中安"等）- サーバー側で自動生成
plate_result_id         # 打席結果マスタID
batting_position_id     # 守備位置（旧）
hit_direction_id        # 打球方向マスタID（13方向）
created_at, updated_at

# 追加カラム（打席結果の詳細）
out_type                 # アウト種別: ゴロ/フライ/ライナー/併殺打/ファールフライ
hit_type                 # ヒット種別: 単打/二塁打/三塁打/本塁打
rbi                      # 打点
run_scored               # 得点（自身が得点したか）integer
stolen_bases             # 盗塁数 integer
caught_stealing          # 盗塁死数 integer

# 追加カラム（打席状況：任意）
final_balls              # 最終ボールカウント (0-3)
final_strikes            # 最終ストライクカウント (0-2)
final_outs               # 打席時のアウトカウント (0-2)
first_pitch_swing        # 初球打ちフラグ (boolean)
runners_state            # ランナー状況 (enum 0-7)
inning                   # 何回 (integer, 任意)

# 追加カラム（打席詳細：任意）
contact_quality_id       # 打球の質マスタID
timing_id                # タイミングマスタID
pitch_type_id            # 最後に打った球種マスタID
self_analysis_memo       # 自己分析メモ (text)
opponent_memo            # 対戦相手・配球メモ (text)

# 追加カラム（打球方向：絶対位置）
hit_location_x           # 正規化座標 (decimal 0.0-1.0)
hit_location_y           # 正規化座標 (decimal 0.0-1.0)
```

**`runners_state` の enum 定義**:
| 値 | 状況 |
|---|------|
| 0 | 無走者 |
| 1 | 一塁 |
| 2 | 二塁 |
| 3 | 三塁 |
| 4 | 一二塁 |
| 5 | 一三塁 |
| 6 | 二三塁 |
| 7 | 満塁 |

得点圏判定: `runners_state IN (2, 3, 4, 5, 6, 7)` （二塁・三塁・一二塁・一三塁・二三塁・満塁）

**注意**: `out_type` の選択肢に「ファールフライ」を含める（既存 `plate_result_id = 3` ファールフライとは別系統。新仕様では `plate_result_id = アウト` + `out_type = ファールフライ` で記録）。

#### `match_results` テーブル

```
stadium_id  # 球場マスタID（FK、任意 / nullable）
```

#### `batting_average` テーブル

- 構造変更なし
- 値の生成方法を変更: `plate_appearances` の after_commit で自動再計算

### 3.2 新規マスタテーブル

#### `stadiums`（球場マスタ）

```
id
name                  # 球場名
prefecture_id         # 都道府県（任意、絞り込み用）
created_by_user_id    # ユーザー追加分の追跡（任意）
created_at, updated_at
```

運用: チームマスタと同様。既存マスタから検索 → なければユーザー追加可能（admin レビューなし）。

#### `pitch_types`（球種マスタ）

```
id
name              # 球種名
display_order     # 表示順
```

初期データ:
1. ストレート系
2. ツーシーム系
3. カット系
4. シュート系
5. スライダー系
6. カーブ系
7. シンカー系
8. フォーク系
9. スプリット系
10. チェンジアップ系

#### `contact_qualities`（打球の質）

```
id, name, display_order
```

初期データ: 真芯 / 先っぽ / 詰まり / 擦り / ドライブ

#### `timings`（タイミング）

```
id, name, display_order
```

初期データ: ドンピシャ / 泳ぎ気味 / 遅れ気味

### 3.3 既存マスタの扱い

#### 打球方向（13方向、コード定数で管理）

- 13方向（投/捕/一/二/三/遊/左線/左/左中/中/右中/右/右線）の id→name は `Stats::HitDirectionAggregator::DIRECTION_LABELS` を SSoT として管理する（DB マスタテーブルは持たない）
- mobile はタップ座標から `mobile/constants/groundCanvas.ts` の `DIRECTION_LABEL_POSITIONS` との距離計算で `hit_direction_id` を導出する
- `plate_appearances.hit_direction_id` は素の integer 列で保存

#### `plate_results`（既存）

- 既存IDをすべて維持（走塁妨害=18, 併殺打=19, ファールフライ=3 などもそのまま）
- 新仕様の入力フォームでは以下の選択肢を **非表示** にする:
  - 走塁妨害（ID 18）
  - 併殺打（ID 19）→ 新仕様では `アウト + out_type = 併殺打` で記録
  - ファールフライ（ID 3）→ 新仕様では `アウト + out_type = ファールフライ` で記録

## 4. 入力フロー

### 4.1 試合記録の3パターン分岐

試合基本情報入力後、以下の3ボタンを表示する:

| ボタン | フロー |
|--------|--------|
| 打撃結果のみ入力 | 打撃結果記録フォーム → 試合結果まとめ画面 |
| 投手結果のみ入力 | 投手結果記録フォーム → 試合結果まとめ画面 |
| 打撃・投手記録を入力 | 打撃結果記録フォーム → 投手結果記録フォーム → 試合結果まとめ画面 |

**判定方法**: DBに専用フラグは持たず、データの有無（`plate_appearances` or `pitching_results` の存在）で判定する。

### 4.2 球場記録

- 試合基本情報入力時に **任意項目** として記録
- 既存マスタから検索 → ヒットしなければユーザー追加可能

### 4.3 打撃結果記録（ステップ式UI）

#### ステップ1: グラウンドイラスト表示 + 打席結果選択

画面構成:
- 上部: グラウンドイラスト（タップで打球方向の絶対位置を記録）
- 下部: 打席結果の選択肢

#### ステップ2: 打球方向タップ状態による分岐

##### 打球方向 未タップ状態

選択肢:
- 空振り三振
- 見逃し三振
- 振り逃げ
- 四球
- 死球
- 打撃妨害

選択後 → ステップ3（+/- 入力）へ

##### 打球方向 タップ済み状態

選択肢:
- アウト → サブ選択: ゴロ / フライ / ライナー / 併殺打 / ファールフライ
- ヒット → サブ選択: 単打 / 二塁打 / 三塁打 / 本塁打
- 失策
- フィルダースチョイス
- 犠打（送りバント）
- 犠飛

選択後 → ステップ3（+/- 入力）へ

#### ステップ3: 打点・得点・盗塁・盗塁死の入力

各項目を `+` / `-` アイコンで増減入力。

#### ステップ4（任意）: 詳細データ入力

スキップ可能であることをUIで明示する:
- 画面ヘッダー: "詳細データを記録（すべて任意）"
- 最下部に「あとで入力する → 完了」ボタンを配置

入力項目:
- 最終カウント（B-S-O）
- 初球打ちフラグ
- ランナー状況
- 打球の質（真芯 / 先っぽ / 詰まり / 擦り / ドライブ）
- タイミング（ドンピシャ / 泳ぎ気味 / 遅れ気味）
- 球種（10種）
- 自己分析メモ（テキスト）
- 対戦相手メモ（テキスト）
- イニング

#### ステップ5: 完了

打席を保存して打撃記録フォームに戻る（次の打席へ進めるリストUIに）。

### 4.4 打席ごとの保存・編集

- **1打席完了ボタンでAPI送信**（リアルタイム保存）
- キャンセル時はローカル破棄（API送信しない）
- 打席ウィザード画面に **打席リスト** を表示し、各打席カードをタップで編集・削除可能
- 順番入れ替えはなし（`batter_box_number` で固定）
- 詳細データ未入力の打席には **「詳細未入力」バッジ** を表示
- 試合終了後も、試合詳細画面の打席カードから詳細データを追記可能

`batting_result` カラムはサーバー側で `plate_result_id` + `out_type/hit_type` + `hit_direction_id` から生成して保存する。フロント側の表示ロジックを削減し、APIレスポンスでそのまま表示可能にする。

例: `中安`、`左二`、`三ゴロ`、`左本`、`犠飛`

## 6. 打球方向判定と座標系

### 6.1 判定方法

- 打球位置は正規化座標 (`hit_location_x`, `hit_location_y`) で保存
- mobile はタップ座標と `DIRECTION_LABEL_POSITIONS` (13方向のラベル中心)との距離計算で `hit_direction_id` (1〜13) を導出する

### 6.2 座標系

- 正規化座標 (0.0〜1.0) で保存（mobile キャンバス 420×340 px を基準にスケール）
- 球場イラスト差し替え時の互換性を担保

| # | 機能 | 内容 | プラン |

#### E. 球種対応分析

| # | 機能 | 内容 | プラン |
|---|------|------|--------|
| E-1 | 球種別打率 | 10種類の球種ごとの打率・長打率 | Pro |

#### F. イニング・球場系

| # | 機能 | 内容 | プラン |
|---|------|------|--------|
| F-1 | イニング別打率 | 回別の打率推移 | Pro |

- **ファーストストライク打率**: `final_strikes = 0` だけでは厳密判定できず、追加フラグも入れない方針のため削除

## 9. マイグレーション戦略

### 9.1 マイグレーション順序

1. **新規マスタテーブル作成**: `stadiums`, `pitch_types`, `contact_qualities`, `timings`
2. **初期マスタデータ投入**: 各マスタの初期値（seed or migration）
3. **`plate_appearances` に新カラム追加**（すべて nullable）
4. **`match_results` に `stadium_id` 追加**（nullable）
5. **`batting_average` の after_commit 自動再計算ロジック追加**

### 9.2 既存データへの影響

- 既存 `plate_appearances` レコードは新カラム NULL のまま
- 既存 `batting_average` レコードはそのまま使用
- 新規入力からは新カラムを使用、`batting_average` は自動再計算で更新
- 既存マスタID（走塁妨害=18 など）はすべて維持
- 既存 `batting_result`（文字列）も維持

### 9.3 段階的移行

- 旧画面は当面残す（フィーチャーフラグまたは新画面への完全切り替え）
- 既存ユーザーの過去試合は「集計値あり・打席詳細なし」状態として残る
- 新仕様で記録された試合は「打席詳細あり」として表示分岐

### 9.4 既存データ保全の検証（必須）

**前提**: 既存ユーザーの成績・集計データ（`game_results` / `plate_appearances` / `batting_average` / `pitching_result` / `match_results` / 各種マスタ）が **ずれない / 消えない / 改変されない** ことを、各実装フェーズで **必ず検証しながら進める**。

#### 検証チェックリスト

各 PR/ブランチで以下を実施:

- [ ] **マイグレーション前後で件数差分ゼロ**: `plate_appearances`, `batting_average`, `pitching_result`, `match_results`, `game_results` の各テーブルでレコード件数が変わらない
- [ ] **既存集計値の不変性**: 既存ユーザーの `batting_average` の主要カラム（`at_bats`, `hit`, `two_base_hit`, `three_base_hit`, `home_run`, `total_bases`, `runs_batted_in`, `strike_out`, `base_on_balls`, `hit_by_pitch`, `sacrifice_hit`, `sacrifice_fly`, `stealing_base`, `caught_stealing`, `error`）の値がマイグレーション前後で変わらない
- [ ] **既存マスタIDの保持**: `plate_results`, `batting_positions` の既存IDがすべて維持されている（走塁妨害=18, 併殺打=19, ファールフライ=3 など）。打球方向 (1〜13) は `Stats::HitDirectionAggregator::DIRECTION_LABELS` 定数で保持
- [ ] **既存表示の変化なし**: 既存試合（打席詳細なし）が試合一覧・試合詳細・ランキングで従来通り表示される（特に `batting_result` 文字列カラムの既存値）
- [ ] **after_commit 自動再計算の安全性**: 新仕様で打席保存時に `batting_average` を再計算する際、既存試合（打席詳細なし）の集計値は **再計算対象から除外** する（または再計算しても同じ値になることを保証する）
- [ ] **ロールバック可能性**: マイグレーションは `db:rollback` で安全に戻せる（カラム追加・マスタテーブル追加のみで破壊的変更なし）

#### 検証手段

- **マイグレーション前後のスナップショット比較**: 開発環境で実データ相当の seed を流し、マイグレーション実行前後で対象テーブルの集計値をダンプして diff を取る
- **RSpec での既存データ保全テスト**: マイグレーション後にも既存ユーザーの集計値が変わらないことを spec で担保
- **本番リリース前のステージング確認**: ステージング環境で本番DBクローンに対してマイグレーションを試行し、上記チェックリストを実施
- **Sentry監視**: リリース後 24-48 時間は `batting_average` / `plate_appearances` 関連のエラーを重点監視

#### 影響範囲別の注意点

| 触る対象 | 想定リスク | 検証ポイント |
|---------|-----------|-------------|
| `plate_appearances` カラム追加 | 既存レコードのカラム値破壊 | 新カラムはすべて nullable で追加、デフォルト値で既存値書換しない |
| `batting_average` 再計算ロジック | 既存集計値の改変 | 既存試合（打席詳細なし）は再計算対象から除外、新仕様試合のみ対象 |
| 既存 `plate_result_id` の選択肢非表示 | 既存データの表示崩れ | 既存IDは残し、フォームUI上でのみ非表示制御 |
| `match_results` に `stadium_id` 追加 | 既存試合の必須化トラブル | nullable で追加、既存試合は NULL のまま |
